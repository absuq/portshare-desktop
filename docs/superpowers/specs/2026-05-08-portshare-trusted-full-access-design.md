# portshare 可信设备全端口访问设计

## 背景

当前直连配对已经能证明两台 `portshare` 通过 Tailscale IP 和共享密钥互相信任，但配对成功后并不会改变 Windows 入站访问策略。因此用户仍可能遇到：Tailscale ping 成功、某些端口可访问、某些端口不可访问。

本阶段目标是让配对后的两台电脑具备更接近局域网内设备的能力：只要一端有服务监听在可被 Tailscale IP 访问的地址上，另一端就可以通过对端 Tailscale IP 访问该服务。

## 目标

- 配对成功后，自动为可信设备授予本机全端口入站访问权限。
- 授权范围限制为对方 Tailscale IP，不能向普通局域网或公网开放。
- 授权 TCP 和 UDP 的全部本地端口。
- 发起方配对成功时授权对方访问本机。
- 响应方接受有效握手时也授权发起方访问本机。
- UI 明确显示“已配对并授权全端口访问”。
- 授权失败时给出可操作提示，尤其是需要管理员权限时。

## 非目标

- 不实现业务端口转发。
- 不代理 TCP/UDP 流量。
- 不绕过 Windows 防火墙以外的服务绑定限制。
- 不强制修改 Tailscale ACL。
- 不保证没有服务监听的端口可以连通。
- 不把任意端口暴露到公网。

## 语义

“全端口访问”表示：

- 本机 Windows 防火墙允许指定可信设备的 Tailscale IP 入站访问本机 TCP/UDP 任意端口。
- 如果服务只监听 `127.0.0.1`，对方仍不能通过 Tailscale IP 访问它。
- 如果目标端口没有进程监听，对方仍会看到连接失败。
- 如果 Tailscale ACL 禁止两台设备互访，`portshare` 不能越过 ACL。

这与真实局域网类似：网络路径和防火墙允许访问，但端口是否成功取决于目标机器是否有服务监听在可访问地址上。

## Windows 防火墙规则

每个可信设备创建两条入站允许规则：

- TCP：允许 `remoteip=<peer-tailscale-ip>` 访问 `localip=<local-tailscale-ip>` 的任意 TCP 端口。
- UDP：允许 `remoteip=<peer-tailscale-ip>` 访问 `localip=<local-tailscale-ip>` 的任意 UDP 端口。

规则名称稳定，包含 `portshare`、对端设备标识、对端 Tailscale IP 和协议。重复配对时先删除旧同名规则，再创建新规则，保证幂等。

写入 Windows 防火墙规则需要管理员权限。当前 MVP 不自动提权；如果写入失败，UI 提示用户以管理员身份运行 `portshare` 后重试配对。

## 双向授权流程

发起方：

1. 输入共享密钥并启用直连监听。
2. 输入对方 Tailscale IP 并点击配对。
3. 握手成功。
4. 保存对方为可信设备。
5. 写入允许对方 Tailscale IP 访问本机全端口的防火墙规则。
6. UI 显示已配对并授权。

响应方：

1. 输入同一个共享密钥并启用直连监听。
2. 收到对方握手请求。
3. 验证共享密钥证明。
4. 保存发起方为可信设备。
5. 写入允许发起方 Tailscale IP 访问本机全端口的防火墙规则。

## 错误处理

- 握手失败：不保存可信设备，不写入防火墙规则。
- 防火墙授权失败：配对操作返回失败，提示需要管理员权限或检查 Windows 防火墙服务。
- 无法识别本机 Tailscale IP：不写入宽泛规则，提示先检测 Tailscale 并启用直连监听。
- 无法识别对方 Tailscale IP：不写入规则。

## 测试策略

- 单元测试规则生成：TCP/UDP 两条规则、远端 IP、本地 IP、稳定规则名。
- 单元测试命令执行：先删除旧规则，再添加 TCP/UDP 规则。
- manager 测试：发起方配对成功后调用授权器并保存授权时间。
- server 回调测试：响应方认证成功后调用 manager 的保存和授权逻辑。
- UI controller 测试：配对成功文案包含“授权全端口访问”。
- 全量 `go test ./...`、`go vet ./...` 和 Windows 构建。
